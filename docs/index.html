Internal Technical Reference

SKYTRACK Ground Control Station

A real-time UAV ground control system. Live telemetry on a 3D globe, live drone camera video, and a military C2 dashboard — all running in Docker with a single command.

Spring Boot 3.2 React 18 + Vite CesiumJS PostgreSQL MediaMTX STOMP / WebSocket MAVLink UDP HLS Video JWT Auth Docker
Section 01

What is Sherlock GCS?

Sherlock GCS (Ground Control Station) is a defense-style real-time dashboard for monitoring an Unmanned Aerial Vehicle (UAV / drone). Think of it as a mission control screen: you see the drone moving on a live 3D globe, its altitude, speed, battery, heading, and optionally its live camera feed — all updating twice per second.

There is no real drone. The backend simulates one in software, broadcasting fake but realistic telemetry data. For the camera feed you use FFmpeg to stream a video file (or a synthetic pattern) into a media server called MediaMTX, which the frontend then plays back via HLS.

The five services

1
PostgreSQL — the database

Stores every telemetry snapshot forever. Spring Boot writes to it; the frontend reads the last 150 points to draw the flight path on startup.

2
Backend (Spring Boot) — the brain

Simulates a drone at 2 Hz, pushes telemetry over WebSocket to all connected browsers, and persists it to Postgres. Also exposes REST endpoints for history and stream URL resolution.

3
Frontend (React) — the dashboard

A React app that subscribes to the backend's WebSocket, renders the drone on a CesiumJS 3D globe, and shows a military C2-style telemetry panel. Two Docker modes: prod serves a static build via nginx on :80; dev runs the Vite dev server with HMR on :5173.

4
MediaMTX — the video server

Accepts an RTSP video stream pushed by FFmpeg (your drone camera simulator) and re-serves it as HLS. The frontend plays the HLS stream in the floating Live Video window.

5
FFmpeg — the drone camera sim

Not a Docker service — you run it manually on your host once the stack is up. It reads a video file (or generates a test pattern) and pushes it as RTSP to MediaMTX. This simulates what a real drone camera would do.

Section 02

Running the Stack

Prerequisites

ToolWhy you need it
Docker + Docker ComposeRuns all four services (Postgres, backend, frontend, MediaMTX)
FFmpegSimulates the drone camera feed (pushed into MediaMTX)
A browserOpens the dashboard — any Chromium-based browser works best

Step 1 — Copy the environment file

The repo ships with a .env.example. Docker Compose reads from .env, so copy it first: 

cp .env.example .env

The file ships with development defaults including a seed operator (admin / sherlock) that is created automatically on first startup — no manual SQL needed for local work.

In any non-development deployment you must replace JWT_SECRET with a random 512-bit key and clear DEV_SEED_USER / DEV_SEED_PASSWORD. Generate a production secret with: 

openssl rand -base64 64

Step 2 — Start everything

The stack uses Docker Compose profiles to select the frontend mode. Pick one depending on what you need: 

# Development — Vite dev server with HMR on :5173 (edit files, browser updates instantly)
docker compose --profile dev up --build

# Production — nginx static build on :80
docker compose --profile prod up --build

# Subsequent runs (no rebuild needed unless code changed)
docker compose --profile dev up

Wait until you see the Spring Boot banner and Started SherlockApplication in the logs. PostgreSQL must be healthy before the backend starts (Docker handles this automatically).

Step 3 — Open the dashboard

Navigate to the URL that matches your chosen profile:
Dev profile: http://localhost:5173
Prod profile: http://localhost

You will see the Operator Authentication login screen first. Enter the credentials for an account that has been added to the operators database table. See Section 03 — Authentication for how to create accounts.

Tip — Running services outside Docker

You can also run each service directly without Docker for maximum iteration speed:

Backend: cd backend && mvn spring-boot:run (needs local Postgres)
Frontend: cd frontend && npm install && npm run dev → opens at http://localhost:5173

Section 03

Authentication

Every part of Sherlock GCS — the REST API, the WebSocket, and the React UI — is gated behind a JWT login. There is no sign-up screen. Operator accounts are created exclusively by inserting a row directly into the database.

Logging in

On first load the app shows a full-screen Operator Authentication form. Enter a valid Operator ID and Access Code. A successful login returns a JWT that is stored in sessionStorage and automatically attached to every subsequent API call and WebSocket connection. The token expires after 8 hours (one operator shift) — the operator must log in again after that. Closing the browser tab also clears the session.

Creating operator accounts

Development — automatic seed user

Set DEV_SEED_USER and DEV_SEED_PASSWORD in your .env file before starting the stack. DevDataInitializer creates the operator on startup and skips silently if the username already exists. .env.example ships with admin / sherlock as ready-to-use defaults — no SQL required. 

# .env — development only
DEV_SEED_USER=admin
DEV_SEED_PASSWORD=sherlock

Production — direct database insert

Leave DEV_SEED_USER and DEV_SEED_PASSWORD unset. Insert directly into the operators table with a BCrypt cost-12 password hash. 

-- Run in psql or DBeaver against the sherlock database
INSERT INTO operators (id, username, password_hash, is_enabled, failed_attempts, created_at)
VALUES (
  gen_random_uuid(),
  'operator1',
  '$2a$12$<bcrypt-hash-of-the-password>',
  true,
  0,
  now()
);

To generate the BCrypt hash, use any cost-12 BCrypt tool, or call new BCryptPasswordEncoder(12).encode("yourpassword") in a temporary backend main method.

Account lockout

After 5 consecutive failed login attempts an account is automatically locked for 30 minutes. The operator will see "Account temporarily locked — contact your system administrator". To unlock early, set failed_attempts = 0 and locked_until = NULL directly in the database. 

UPDATE operators
SET failed_attempts = 0, locked_until = NULL
WHERE username = 'operator1';

Audit log

Every login attempt — success or failure — is written to the auth_audit_log table with a timestamp, IP address, and failure reason. This table is append-only and never modified.

Production requirement

The default JWT_SECRET in .env.example is a development placeholder. It must be replaced with a unique random key in any environment that handles real mission data. Generate one with openssl rand -base64 64.

Section 04

How Telemetry Data Flows

Every 500 ms the backend's TelemetrySimulator fires. Here is exactly what happens each tick:

1
Simulator fires (Java, every 500 ms)

Advances latitude/longitude using great-circle maths, oscillates altitude, drains battery ~3%/min, drifts heading ±3°. All state lives in instance fields on one thread — intentionally simple.

2
Broadcast over STOMP / WebSocket

The new TelemetryDTO is serialised to JSON and sent to /topic/telemetry/{droneId} for selected-drone detail. In parallel, a fleet-lite snapshot is sent to /topic/telemetry/lite/fleet for all-assets rendering.

3
Persisted to PostgreSQL

Same data is written to the telemetry table via JPA. This gives you full history even after the browser refreshes.

4
Frontend receives via STOMP client

The useTelemetry hook in React connects to the WebSocket with a JWT in the STOMP CONNECT headers. The server validates the token before the subscription is granted. Each message is then validated and stored in React state — the globe moves, panels update.

5
Bulk last-known bootstrap on startup

On login, the frontend calls POST /api/telemetry/last-known once with the fleet IDs. This removes the per-drone request burst and fills the asset selector with last known coordinates in one response.

Section 05

Live Video — How It Works

The live video pipeline is deliberately separated from telemetry. The backend never touches video bytes — it only tells the frontend where the video lives. MediaMTX handles all of the streaming heavy-lifting.

End-to-end pipeline

1
Video source pushes RTSP to MediaMTX

A real drone (or FFmpeg on your laptop) pushes an H.264 video stream over RTSP to rtsp://localhost:8554/sherlock-01. MediaMTX accepts it and automatically creates the path — no config needed.

2
User clicks SETTINGS → LIVE VIDEO: OFF

The toggle is inside the SETTINGS dropdown in the top-right of the header (only visible when a drone is selected). Clicking it sets the state to ON.

3
Frontend fetches the stream URL

useStreamUrl calls GET /api/drones/SHERLOCK-01/stream. The backend returns { "streamUrl": "http://mediamtx:8888/sherlock-01/index.m3u8" }.

4
LiveVideoWindow mounts + hls.js plays

A 240×240 floating window appears in the bottom-right of the map area. hls.js loads the index.m3u8 manifest and begins buffering and playing the video segments. On Safari, native HLS is used instead.

5
Video goes through the nginx /hls/ proxy

In Docker the browser actually requests http://localhost/hls/sherlock-01/…. Nginx proxies this to mediamtx:8888 inside the Docker network — the browser never needs direct access to MediaMTX.

Important — video has latency

HLS works by slicing the stream into small segments (usually 2–6 seconds each). This means the live feed will be roughly 5–15 seconds behind real-time. This is normal and acceptable for a GCS operator view. If you need sub-second latency in the future, the WebRTC port (:8889) is already exposed on MediaMTX for that upgrade path.

Section 06

Simulating the Drone Camera (Development)

Since you don't have a physical drone, FFmpeg acts as the camera. It pushes a video stream into MediaMTX exactly the same way a real drone would. There are two approaches:

Option A — Loop a real video file

Use any .mp4 file — drone footage from YouTube, a GoPro clip, anything. The -stream_loop -1 flag loops it indefinitely. 

ffmpeg \
  -re \
  -stream_loop -1 \
  -i your_footage.mp4 \
  -c:v libx264 \
  -preset ultrafast \
  -tune zerolatency \
  -f rtsp \
  rtsp://localhost:8554/sherlock-01

Option B — Synthetic test pattern (no video file needed)

FFmpeg can generate a test card with an animated text overlay purely in software. No input file required at all. 

ffmpeg \
  -re \
  -f lavfi \
  -i "testsrc2=size=1280x720:rate=30,drawtext=text='DRONE CAM SIM':fontsize=36:fontcolor=lime" \
  -c:v libx264 \
  -preset ultrafast \
  -tune zerolatency \
  -f rtsp \
  rtsp://localhost:8554/sherlock-01
Why sherlock-01?

The drone ID in the app is SHERLOCK-01. The backend lowercases it to build the MediaMTX path, so the RTSP path must be sherlock-01 (all lowercase, hyphen). If you push to a different path, the frontend's HLS URL won't match and the video won't play.

Full development workflow (step by step)

1
Terminal 1 — start the Docker stack
docker compose --profile dev up --build

Wait for Started SherlockApplication. Use --profile prod instead for the nginx build on :80.

2
Terminal 2 — start the camera simulator
ffmpeg -re -stream_loop -1 -i footage.mp4 \
  -c:v libx264 -preset ultrafast -tune zerolatency \
  -f rtsp rtsp://localhost:8554/sherlock-01

You should see FFmpeg printing frame counts. Keep this running.

3
Browser — log in and open the dashboard

Go to http://localhost:5173 (dev profile) or http://localhost (prod profile). Enter your operator credentials on the login screen, then click SHERLOCK-01 on the globe overlay to select the drone.

4
Click SETTINGS → LIVE VIDEO: OFF

The SETTINGS button is in the top-right header, next to the drone name. Click it, then click LIVE VIDEO: OFF to toggle it on. A 240×240 video window will appear over the bottom-right of the map.

5
Wait ~5–10 seconds for the first HLS segment

The window will show ACQUIRING FEED… while MediaMTX buffers the first segments. Then video plays automatically. If you see ⚠ FEED ERROR, check that FFmpeg is still running and pushing to the correct RTSP path.

Section 08

Mission Planning

Mission Planning lets you define a named waypoint route before the flight, save it to the database, and execute it server-side against a real MAVLink drone. The backend drives the entire execution loop — the drone keeps flying even if you close the browser tab.

Requires a real MAVLink drone or ArduPilot SITL

Mission execution requires MAVLINK_ENABLED=true in your .env and a MAVLink drone (or SITL) sending telemetry to udp://localhost:14550. You can still create and save missions at any time — execution will return 503 MAVLINK DISABLED if the adapter is off.

How to create a mission

1
Select a drone and open SETTINGS → MISSION PLAN: OFF

The SETTINGS dropdown is next to the drone name in the header. Clicking MISSION PLAN: OFF toggles it to ON. The right panel switches from System Panel to Mission Planning Panel. Driver Mode is automatically disabled — the two modes are mutually exclusive.

2
Click the map to place waypoints, then adjust with gizmo

Each left-click adds an amber numbered waypoint marker at that position. The waypoint's altitude defaults to the drone's current live altitude. A connecting line joins the waypoints in order. Click a waypoint to select it and drag the XYZ gizmo arrows (X east-west, Y north-south, Z altitude AMSL), or use panel nudge controls. While Mission Planning mode is active, map camera mouse controls are locked to prevent conflict with node/gizmo manipulation. At least 2 waypoints are required before saving.

3
Name and save the mission

In the NEW tab, enter a mission name (e.g. PATROL-ALPHA) and click SAVE. The mission is persisted to the database via POST /api/missions and appears in the SAVED tab with status PLANNED. Click CLEAR to discard the planning waypoints without saving.

4
Edit saved PLANNED missions in place (optional)

In the SAVED tab click EDIT on a PLANNED mission. You can rename, move/add/delete waypoints, and save with PUT /api/missions/{id}. Use CANCEL to discard unsaved edits.

5
Execute the mission

In the SAVED tab, click EXECUTE next to a PLANNED mission. The backend starts a server-side execution loop and the panel switches to the Active Mission view.

Server-side execution — how it works

Once execution starts, the backend's MissionExecutorService takes over. Every 500 ms it:

1
Reads the drone's last-known position

Uses the in-memory telemetry cache — no extra database hit per tick.

2
Dispatches GOTO to the active waypoint

Sends a MAVLink SET_POSITION_TARGET_GLOBAL_INT packet (GUIDED mode) via the same path as the manual GOTO command. The altitude in the mission is stored in AMSL and is automatically converted to relative-home before the MAVLink packet is sent.

3
Checks for waypoint arrival

Uses the haversine formula to compute horizontal distance. Arrival threshold: ≤ 30 m horizontal and ≤ 15 m vertical. When both are met the waypoint is marked REACHED.

4
Advances to the next waypoint (or completes)

The next waypoint in sequence becomes ACTIVE and a new GOTO is sent. When the last waypoint is reached the mission transitions to COMPLETED and the executor removes it from the active queue.

5
Publishes progress via STOMP

On every waypoint status change a full MissionDTO is broadcast to /topic/missions/{id}/progress. The frontend polls GET /api/missions/{id} every second as a fallback — no second STOMP client is needed.

Aborting a mission

Click ABORT MISSION in the Active Mission view (red border button). The backend immediately stops sending GOTO commands, marks all remaining ACTIVE and PENDING waypoints as SKIPPED, and transitions the mission to ABORTED. Section 09

Common Problems

Login button stays disabled / nothing happens

The Operator ID or Access Code field is empty — the button is disabled until both fields have content. If you click AUTHENTICATE and get "Authentication failed", the credentials are wrong. For a default dev setup the credentials are admin / sherlock (set via DEV_SEED_USER / DEV_SEED_PASSWORD in .env). Check the backend logs for the line [DEV] Seed operator 'admin' created to confirm the account was seeded — if it's missing, the env vars weren't set before startup.

Account locked — can't log in

Five consecutive wrong passwords trigger a 30-minute lockout. Wait it out, or unlock early by running this against the database: 

UPDATE operators
SET failed_attempts = 0, locked_until = NULL
WHERE username = 'operator1';

Logged out immediately after logging in

The token expired before it was used — this can happen if the system clock on the server differs significantly from the browser clock. It can also happen if you copied an old token from session storage. Log in fresh; the new token will have a valid 8-hour window from the current server time.

CONNECTING… after login never turns to LINK ACTIVE

The WebSocket connected but the STOMP handshake failed. This usually means the JWT was rejected by the server's channel interceptor — either it expired or the JWT_SECRET in the backend doesn't match the one used to sign the token. Restart the backend after setting the correct secret.

Dashboard is blank / no globe

The backend probably hasn't started yet, or Postgres isn't healthy. Check docker compose logs backend. Wait for Started SherlockApplication.

CONNECTING… never turns to LINK ACTIVE

The browser can't reach the WebSocket. Make sure the URL matches your profile: http://localhost:5173 (dev) or http://localhost (prod). Never open localhost:8080 directly — the WebSocket must go through the Vite or nginx proxy.

Live Video shows ⚠ FEED ERROR

The backend returned an error for GET /api/drones/SHERLOCK-01/stream — or MediaMTX isn't running. Check docker compose logs mediamtx.

Live Video shows ACQUIRING FEED… indefinitely

MediaMTX is running but nothing is pushing RTSP to it. Start the FFmpeg command from Section 05. Also confirm the RTSP path is exactly sherlock-01 (lowercase, hyphen).

EXECUTE button is greyed out on a saved mission

The button is disabled when any of these conditions is true: (1) no drone is currently selected, (2) the selected drone is a simulated SHERLOCK- drone rather than a real MAVLINK- drone, or (3) another mission is already ACTIVE. Select a MAVLINK- drone and make sure no other mission is running before executing.

Mission execute returns MAVLINK DISABLED

The backend's MAVLink adapter is not enabled. Set MAVLINK_ENABLED=true in your .env file and restart the backend. You also need an ArduPilot SITL or real drone sending MAVLink UDP to localhost:14550 before commands can be dispatched.

No flight path on the globe

The drone history query returned nothing — Postgres might be empty on a fresh start. Wait 10–20 seconds for the simulator to accumulate some telemetry points, then re-select the drone.

After any docker compose down

Postgres data persists in a named Docker volume (postgres_data). To wipe it completely and start fresh, run docker compose down -v. Without -v your telemetry history survives restarts.
